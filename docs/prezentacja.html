02 / 07

Zrealizowane wymagania PRD

User Stories

  • Kontrahent widzi liste oswiadczen do zlozenia w danym miesiacu/roku (US c, d)
  • Kontrahent wypelnia i wysyla oswiadczenie (US e, f)
  • Potwierdzenie zlozenia oswiadczenia (US g)
  • Podglad zlozonego oswiadczenia (US h)
  • Skladanie korekt oswiadczen (US i)

Funkcje obowiazkowe

  • Dashboard z okresami rozliczeniowymi wymagajacymi obslugi (15.1)
  • Lista oswiadczen rozliczeniowych z podzialem na statusy (15.3)
  • Formularz skladania oswiadczenia z dynamicznymi polami (15.4)
  • Podglad zlozonego oswiadczenia (15.5)
  • Szablony danych wg typu oplaty i kontrahenta (15.6)

Kryteria MVP (Level I)

  • Min. 2 typy uzytkownikow: OSDp Wytworca
  • Lista oswiadczen dla zalogowanego kontrahenta
  • Min. 2 typy oplat: OP OZE

Bezpieczenstwo i logowanie

  • Uwierzytelnianie login + haslo (BCrypt)
  • Sesje HTTP z kontrola dostepu
  • Kontrola dostepu — uzytkownik widzi tylko dane swojego kontrahenta
  • Trzy role: Administrator, Kontrahent OSDp, Kontrahent Wytworca
03 / 07

Zakres realizacji vs PRD

Zrealizowane szablony formularzy

  • Oplata przejsciowa (OP) — OSDp: 9 pol (gospodarstwa, moce, zuzycie)
  • Oplata OZE — OSDp: 4 pola (nalezna kwota, nieisciagalnosc, energia)
  • Oplata OZE — Wytworca: 1 pole (planowana ilosc energii)

Numeracja oswiadczen

  • Format: OSW/{typ_oplaty}/{kontrahent}/{rok}/{miesiac}/{podokres}/{wersja}
  • Sufiks /KOR dla korekt (po terminie)
  • Automatyczna inkrementacja wersji

Swiadomie poza zakresem MVP

  • Panel administracyjny (zarzadzanie uzytkownikami, szablonami)
  • Harmonogram rozliczen (widok kalendarza)
  • Typy kontrahentow: OSDn, OK, Magazyn
  • Oplaty przesylowe (OSS, OZS, OJ, OR)
  • Oplaty: OKO, OM
  • Edycja danych uzytkownika przez administratora
  • Filtrowanie i sortowanie kolumn w tabelach
  • Rozwinieta sekcja z lista zlozonych oswiadczen
  • Status "Robocze" (zapisz bez wysylania)
Swiadomie wybralismy realizacje uproszczonej funkcjonalnosci spelnijacej kryteria MVP (Level I), zamiast probowac pokryc caly zakres PRD.
04 / 07

Dodatkowe funkcjonalnosci

Funkcje wykraczajace poza wymagania PRD, zaproponowane i zaimplementowane przez Claude Code

Wersjonowanie oswiadczen

  • Automatyczna inkrementacja numeru wersji przy ponownym zlozeniu
  • Widok historii wersji dla danego okresu rozliczeniowego
  • Automatyczne wykrywanie korekty (zlozenie po terminie)
  • Oznaczenie /KOR w numerze oswiadczenia

Eksport JSON do bazy

  • Serializacja calego oswiadczenia do JSON
  • Zapis w kolumnie CLOB w bazie Oracle
  • Gotowy format do integracji z systemami zewnetrznymi

Zaawansowany dashboard

  • Pogrupowanie okresow wg typu oplaty z przelacznikami
  • Paginacja (5 elementow na strone)
  • Licznik brakujacych oswiadczen per typ oplaty
  • Wizualne oznaczenie przekroczonych terminow
  • Wyswietlanie numeru wersji i statusu korekty

Infrastruktura

  • Pelna konteneryzacja (Docker Compose) — 3 serwisy
  • Multi-stage build frontendu (Node + nginx)
  • Seed data: 18 okresow rozliczeniowych (6 miesiecy, rozne podolkresy)
  • Testy jednostkowe (Vitest) i E2E (Playwright)
  • Global exception handler z ustandaryzowanymi odpowiedziami
05 / 07

Zalozenia i podejscie

Wybor technologii

  • Wybralismy technologie, w ktorych nie programujemy na co dzien
  • Cel: sprawdzenie jak AI radzi sobie z nieznanym nam stosem
  • Kotlin + React + Oracle — kazda warstwa to nowe terytorium

Konfiguracja AI

  • Nie modyfikowalismy pliku PRD dostarczonego przez biznes
  • Nie konfigurowalismy Claude (brak CLAUDE.md na starcie)
  • Jedna sesja Claude Code na wszystkie warstwy
  • Praca na monorepo (frontend + backend + baza)

Sposob pracy z AI

  • Zdefiniowalismy uproszczona funkcjonalnosc razem z Claude w planning mode
  • Kolejne iteracje realizowane za pomoca pojedynczych promptow
  • Proste prompty opisujace funkcjonalnosc bez szczegolowej implementacji
  • Pozwolilismy Claude'owi podejmowac decyzje architektoniczne

Strategia realizacji

Krok 1
Przekazanie PRD do Claude w planning mode
Krok 2
Wspolne zdefiniowanie zakresu MVP
Krok 3
Iteracyjne prompty: "dodaj logowanie", "zrob dashboard", "dodaj formularz"...
Krok 4
Poprawki i dopieszczanie UI

07 / 07

Wnioski i przemyslenia

Droga do obecnego stanu

  • Startowalismy z zerowa wiedza o korzystaniu z agentow AI do kodowania
  • Pierwsza proba: realizacja w Oracle APEX — wydajnosciowo nie sprawdzilo sie na maszynie wirtualnej
  • Praca od zera: wybor technologii, konfiguracja srodowiska, wybor AI do stanu obecnego — 7 godzin

Co nas zaskoczylo

  • Zaskakujace rozumienie naszych instrukcji przez AI — proste prompty wystarczyly
  • Claude samodzielnie zaproponowal architekture, konfiguracje Dockera, seed data
  • Generowanie calych warstw aplikacji na podstawie krotkiego opisu funkcjonalnosci
  • Trudno bedzie wrocic do zmudnego kodowania recznego

Wyzwania

  • Nie mozna robic poprawek bez kontekstu Claude — trzeba wrocic do sesji
  • Pozwolilismy Claude'owi "isc na zywiol" — poczucie braku kontroli nad generowanym kodem
  • Przy nastepnych zastosowaniach bedziemy go nieco bardziej pilnowac

Kluczowe lekcje

  • Planning mode to klucz — warto zdefiniowac zakres przed kodowaniem
  • Proste prompty > szczegolowe specyfikacje techniczne
  • Monorepo + jedna sesja = spojnosc miedzy warstwami
  • AI radzi sobie z nieznanym nam stosem technologicznym
Od zera do dzialajacego full-stacka w 7 godzin — bez doswiadczenia z AI coding agents i w technologiach, ktorych nie znamy.
